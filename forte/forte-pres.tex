\begin{frame}{Usual practice: Top-Down Approach}
\textbf{Steps:}
\begin{enumerate}
  \item Write the global specification
  \item Project to obtain the local specifications
  \item Write local programs that respect the specifications
\end{enumerate}
\textbf{Problem:} difficult to integrate existing code and architecture.
\end{frame}
\note{
This approach is very useful because it can assure 
some properties over the communication.
However, with this approach it's difficult to integrate
existing or legacy code.
}

% slide 10
\begin{frame}{Bottom-up approach}
\begin{block}{Extraction steps}
\begin{enumerate}
  \item Analyze an input source code
  \item Extract the local views
  \item Compose local views to create an approximated global view
\end{enumerate}
\end{block}
\textbf{Output:} an abstraction that captures all the possible behaviors.
% \textbf{Over-approximation:} explore all possible behaviors 
% (including some that may not occur).
% in order to ensure no potential issues are missed
\end{frame}
\note{
With this approach, we obtain an abstraction that should capture all
the possible behavior of the system, just by analyzing the code.
And with this abstraction, we can perform some analysis over it to
discover some properties.
}


% slide 11
\begin{frame}{Why}
\begin{block}{Possible benefits}
\begin{itemize}
  \item Improves understanding of the code
  \begin{itemize}
    \item Give all good behavior
  \end{itemize}
  \item Highlight bugs
  \begin{itemize}
    \item Like deadlock, race condition, etc...
  \end{itemize}
  % \item Educational potential
\end{itemize}
\end{block}
\end{frame}
\note{
Why, in out opinion, is this useful?
First, we get a better understanding of the code, thanks of the
rap presentation of all the good behaviors.
We can also highlight bugs like deadlocks or race
conditions.
}


% slide 12
\begin{frame}{Requirements}
\begin{itemize}
  \item Automatic extraction
  \item Target mainstream languages
  \item Support creation and removal of participants
  \item Capture good behaviors and highlight misbehaviors
  % vogliamo catturare bad beha senza crashare
  \item Use a simple notation as output
\end{itemize}

\note{
To make this useful in practice, we need several things:
First, the process should be automatic,
using static analysis.
It should highlight both good and bad behaviors,
support multiple actors,
and be applicable to mainstream languages.
Finally, the output should be easy to read,
something people can reason with.
}

\begin{frame}{Problems - Undecidability}
It would imply deciding termination. Choice of:
\begin{itemize}
  \item Under-approximation: we can leave out some useful behavior
  \item \textbf{Over-approximation}: we include many behaviors (good or bad, present or not)
\end{itemize}
\end{frame}
\note{
This comes obviously with some challenges.
One is undecidability. We should be able to decide termination, 
but this is a known undecidable problem. Therefore, we must use some
approximation techniques.
We choose over-approximation over under-approximation because 
it's coherent with the requirements that we established.
}

\begin{frame}[fragile]{Deadlock}

\begin{frame}[fragile]{Dining Philosophers example}
\begin{columns}
\column{0.55\textwidth}
\begin{lstlisting}
philosopher(Fork1, Fork2) $\to$
  send req to Fork1,
  receive ack from Fork1,
  send req to Fork2,
  receive ack from Fork2,
  eat(),
  send release to Fork1,
  send release to Fork2,

fork() $\to$
  receive req from Phil,
  send ack to Phil,
  receive release from Phil,
  fork().
